\begin{abstract}
We build and evaluate \modelname{}, an agentic assistant for day-ahead EV charging scheduling using real Caltech ACN-Data sessions. Given session arrival/departure windows, requested energy, per-session power limits, and a 50~kW site cap, the system produces a feasible schedule minimizing time-of-use (TOU) cost. We compare three approaches: (i)~a CVXPY convex optimizer, (ii)~a single-shot GPT-4o LLM baseline, and (iii)~\modelname{}, which uses GPT-4o to call the optimizer as a verified tool and generate grounded explanations. Over 19 benchmark days, \modelname{} exactly matches the optimizer on all metrics. The LLM baseline's lower reported cost is an artifact of under-delivery: $12\times$ more unmet kWh and $6.6\times$ more checker flags, including hard physical-constraint violations that the solver and agent avoid entirely.
\end{abstract}

\section{Introduction and Related Work}
% ============================================================

\paragraph{Motivation.}
At the Caltech ACN-Data JPL site, peak EV demand can reach 200~kW—nearly $4\times$ the 50~kW site cap—meaning unmanaged charging regularly exceeds limits and raises costs under TOU tariffs. A practical scheduler must handle time-varying prices, overlapping session windows, and per-session power limits~\cite{lee2020acn,wang2016survey}. Convex optimization solves this reliably~\cite{lee2020acn}, but requires precisely structured inputs and returns no natural-language output. LLMs accept free-form queries~\cite{brown2020language}, but fail on constrained numerical programs. We investigate whether routing natural-language requests through an LLM to a convex solver can deliver both the reliability of optimization and the accessibility of natural-language interaction.

\paragraph{Task.} Given a natural-language description of a day's sessions, produce $p_{i,t}$ [kW] that (1) satisfies all physical constraints, (2) minimizes TOU cost, and (3) delivers requested energy wherever the site cap allows.

\paragraph{Contributions.}
\begin{itemize}
  \item \textbf{Method:} \modelname{} maps NL requests to optimizer inputs, calls CVXPY, and generates explanations from tool-returned metrics—structurally preventing numeric hallucination.
  \item \textbf{Evaluation:} A shared pipeline runs all three methods on identical session data, scored by the same constraint checker and metrics code.
  \item \textbf{Outcome:} \modelname{} matches the optimizer exactly over 19 days; the LLM baseline's apparent cost savings reflect under-delivery, not genuine optimization.
\end{itemize}

\paragraph{Related work.}
EV scheduling has been studied at large scale with network constraints~\cite{wang2016survey, lee2020acn}. The ReAct framework~\cite{yao2022react} shows tool-augmented LLMs can improve reliability on structured tasks over single-shot generation. We apply these ideas to EV scheduling, delegating all numerical computation to a certified external solver and thereby providing verifiability guarantees not achievable through LLM reasoning alone.

\section{Problem Formulation}
% ============================================================

\paragraph{Notation.}
Sessions $\mathcal{I}$ are scheduled over $\mathcal{T}=\{0,\dots,95\}$ (15-min steps, one day). Each session $i \in \mathcal{I}$ has: arrival $a_i$, departure $d_i$ (half-open window $[a_i,d_i)$), requested energy $E_i^{\text{req}}$ [kWh], and max power $\bar{p}_i$ [kW]. The site has aggregate cap $\bar{P}_t$ [kW] and TOU price $c_t$ [\$/kWh]. Decision variables: power $p_{i,t}$ [kW] and per-session unmet slack $u_i \ge 0$ [kWh].

\paragraph{Core formulation.}
\begin{align}
\min_{\{p_{i,t}\}, \{u_i\}} \;
  & \Delta t \sum_{t} c_t \sum_{i} p_{i,t} + \lambda \sum_{i} u_i \label{eq:obj_ev} \\
\text{s.t.}\quad
  & p_{i,t} = 0, \quad t \notin [a_i, d_i) \quad \text{(availability)} \label{eq:avail}\\
  & 0 \le p_{i,t} \le \bar{p}_i \quad \text{(per-session power)} \label{eq:per_session}\\
  & \textstyle\sum_{i} p_{i,t} \le \bar{P}_t \quad \text{(site cap)} \label{eq:site_cap}\\
  & \Delta t \sum_{t=a_i}^{d_i-1} p_{i,t} + u_i = E_i^{\text{req}} \quad \text{(energy balance)} \label{eq:energy_bal}\\
  & u_i \ge 0 \label{eq:unmet_nonneg}
\end{align}
With $\Delta t=0.25$~h and $\lambda=10^6$~\$/kWh, the program is linear (hence convex) and solved to global optimality by CVXPY (default solver backend). The large penalty ensures $u_i>0$ only when the site cap makes full delivery physically impossible. All constraints are independently re-verified by the shared constraint checker after solving. The checker reports two categories of flags: \emph{hard constraint violations} on \eqref{eq:avail}--\eqref{eq:site_cap} (physical infeasibility) and \emph{energy shortfall flags} where $u_i > 0$ (under-delivery forced by the site cap).

\paragraph{Success criteria.}
A run is \textbf{successful} if: (1) the schedule has zero hard constraint violations (tol $10^{-5}$) on \eqref{eq:avail}--\eqref{eq:site_cap}. Note energy shortfall flags ($u_i > 0$) are permitted when the site cap makes full delivery physically impossible. (2) Cost, peak, unmet energy, and sessions served match the CVXPY reference on identical data, and (3) every number in the agent's explanation matches the corresponding field in the tool's JSON output.

\section{Methodology}

\subsection{System Overview}

Raw ACN-Data sessions are preprocessed into a standardized table (step-indexed arrival/departure, energy, max power) that feeds three end-to-end pipelines, all sharing the same evaluation layer (Figure~\ref{fig:system_overview}):
\begin{enumerate}
  \item \textbf{Phase~A — Numerical solver:} Direct CVXPY solution of \eqref{eq:obj_ev}--\eqref{eq:unmet_nonneg}. No LLM involved.
  \item \textbf{Phase~B — LLM-only baseline:} A single GPT-4o call receives the full problem as text and outputs the $p_{i,t}$ matrix directly. No optimizer or checker is accessible during generation.
  \item \textbf{Phase~C — \modelname{}:} GPT-4o parses the NL request, calls \tool{solve\_ev\_schedule} (CVXPY wrapper) and constraint checker as OpenAI function-calling tools in a multi-turn loop. Then writes an explanation grounded in the returned metrics.
\end{enumerate}

\subsection{Data, Environment, and Tools}

We use the Caltech ACN-Data JPL site~\cite{acndata} (50~kW cap, 96$\times$15-min steps per day). Raw timestamps are mapped to step indices where sessions are clipped to the day boundary. Missing max-power values filled from charger metadata. Two tools are shared:
\begin{itemize}
  \item \textbf{CVXPY optimizer:} Solves \eqref{eq:obj_ev}--\eqref{eq:unmet_nonneg} with $\lambda=10^6$. Returns optimal schedule, cost, peak, and per-session unmet energy.
  \item \textbf{Constraint checker:} Independently verifies all constraint types (availability, per-session power, site cap, energy over/under-delivery), returns a flag list distinguishing hard constraint violations from energy shortfall flags, and computes all evaluation metrics. Each type is covered by unit tests.
\end{itemize}
In Phase~C the optimizer is exposed via the OpenAI function-calling API whereas in Phase~A it is called directly. In Phase~B neither tool is accessible during generation.

\subsection{Agent Design and Model/Tool Interaction}

\modelname{} follows the ReAct pattern~\cite{yao2022react} in a structured five-step loop (Table~\ref{tab:agent_roles}):
\begin{enumerate}
  \item \textbf{Parse (GPT-4o):} Converts the NL request to structured session parameters—step-indexed arrival/departure windows, per-session energy and max power. Time expressions (e.g., ``6\,pm'') are mapped to step indices (e.g., step~72).
  \item \textbf{Optimize (CVXPY tool):} Calls \tool{solve\_ev\_schedule}, which runs the convex solver and returns a JSON summary: \texttt{total\_cost\_usd}, \texttt{peak\_load\_kw}, \texttt{total\_unmet\_kwh}, \texttt{pct\_fully\_served}.
  \item \textbf{Validate (checker):} The returned schedule is passed to the constraint checker and any flags are surfaced to the agent.
  \item \textbf{Refine (GPT-4o):} On parse error or tool failure, GPT-4o corrects the structured input and retries. The loop completes within 3 rounds on every benchmark query.
  \item \textbf{Explain (GPT-4o):} Writes a natural-language summary using \emph{only} numbers from the tool's JSON output. No metric is inferred by the LLM.
\end{enumerate}

\subsection{Design Rationale and Intellectual Merits}

The central novelty of \modelname{} is the strict separation between language and computation. By routing all arithmetic through certified external tools, we achieve three properties unavailable in a pure LLM approach: (1)~\textbf{Verifiable correctness} — every number in the agent's output is traceable to a specific tool return value, so numeric hallucination is structurally prevented rather than mitigated by prompting. (2)~\textbf{Global optimality} — the agent inherits the convex solver's optimality guarantee without requiring the LLM to reason about optimization. (3)~\textbf{Predictable what-if handling} — scenario queries (e.g., cap reduction, charger offline, extra session) are handled by varying tool inputs rather than rewriting prompts, keeping behavior consistent across query types.

This design is non-trivial in several respects. Prior tool-augmented LLM work (e.g., ReAct~\cite{yao2022react}, Toolformer~\cite{schick2024toolformer}) uses tools primarily for information retrieval or arithmetic. Applying tool-calling to a \emph{constrained convex optimization} problem requires the LLM to correctly translate unconstrained natural-language session descriptions into a fully specified LP — including inferring step indices, max-power bounds, and slack structure — before the solver can be invoked. An error at the parsing stage (e.g., a wrong arrival index or missing session) produces an infeasible or sub-optimal schedule that the checker will flag. The refinement loop (Step~4) handles such failures gracefully, and the shared constraint checker provides objective, post-hoc verification independent of the LLM. This architecture generalizes: any verified solver can replace CVXPY, and any NL front-end can replace GPT-4o, without changing the evaluation pipeline.

\subsection{Metrics}

All methods are evaluated by the shared checker on the final schedule $p_{i,t}$: \textbf{cost} [\$] $= \Delta t\sum_t c_t \sum_i p_{i,t}$; \textbf{peak load} [kW] $= \max_t \sum_i p_{i,t}$; \textbf{unmet energy} [kWh] $= \sum_i\max(0, E_i^{\text{req}}-\Delta t\sum_t p_{i,t})$; \textbf{sessions fully served} [\%]; and \textbf{checker flags} (count), which combine hard constraint violations on \eqref{eq:avail}--\eqref{eq:site_cap} with energy shortfall flags ($u_i > 0$). For the solver and agent, all flags are energy shortfalls on congested days (zero hard violations). For the LLM baseline, flags include hard physical-constraint violations. Cost alone is insufficient: a method can trivially minimize it by delivering no energy, so all metrics must be read jointly.

\subsection{Baselines}

We evaluate three methods under identical inputs and evaluation code. The only controlled variable is tool access during schedule generation:
\begin{itemize}
  \item \textbf{Phase~A — Numerical solver:} CVXPY solves the LP directly. Serves as the gold-standard reference: optimal cost, full constraint compliance, and maximum energy delivery given the site cap.
  \item \textbf{Phase~B — LLM-only baseline:} A single GPT-4o call with the full session description as text outputs the $p_{i,t}$ matrix. No optimizer or checker is accessible. This tests whether a state-of-the-art LLM can solve a constrained LP from natural language without external tools.
  \item \textbf{Phase~C — \modelname{}:} GPT-4o with tool access to the CVXPY optimizer and checker. Tests whether tool-augmented LLM parsing can match the solver from natural-language input.
\end{itemize}
Phases~B and~C receive the same NL session description, site cap, and TOU rates for every day and are scored by the same checker and metrics. This isolation ensures any performance gap reflects tool access rather than input differences, model version, or evaluation bias.

\paragraph{Aggregate results.}
Table~\ref{tab:results} confirms \modelname{} meets all three success criteria on all 19 days. (1)~\textbf{Constraint satisfaction:} zero hard constraint violations on \eqref{eq:avail}--\eqref{eq:site_cap}; the 13.11 average flags are energy shortfalls on physically congested days where the 50~kW cap makes full delivery impossible, not scheduling errors. (2)~\textbf{Optimizer parity:} \$97.19 cost, 47.21~kW peak, 20.95~kWh unmet, 73.86\% served—identical to the CVXPY reference on every day, confirming NL parsing introduces zero scheduling loss. (3)~\textbf{Grounded explanations:} all numerical claims match \tool{solve\_ev\_schedule} JSON. Figure~\ref{fig:perday} visualizes all metrics side-by-side.

The cost structure is instructive: both the solver and \modelname{} achieve cost reduction by \emph{deferring} charging out of the 4--9\,pm peak window (\$0.45/kWh) into off-peak hours (\$0.12/kWh), which is visible as right-shifted power allocation in the schedule heatmaps (Figure~\ref{fig:schedules}). The 73.86\% service rate reflects physically congested days where the cap binds, not scheduling inefficiency — on uncongested days both methods serve 100\% of sessions.

The LLM baseline fails all three criteria. Its \$84.34 average cost is lower only because it delivers far less energy: 251.59~kWh unmet ($12\times$ worse) and 31.83\% sessions served, with 86.11 flags/day ($6.6\times$ more) — including hard constraint violations such as exceeding the site cap. Crucially, the baseline's 60.97~kW average peak \emph{exceeds the 50~kW site cap}, confirming it generates physically infeasible schedules that no real facility could execute.

\paragraph{Per-day analysis and sensitivity.}
The solver and \modelname{} are identical by construction. We observed zero parsing or tool-call errors across all 19 days. The baseline is erratic: on 2019-05-08 it costs \$2.02 with 0\% served (near-zero schedule). On 2019-05-22 it costs \$192.05 at 58.1\% served (overshoots the charge-ASAP baseline cost). This day-to-day inconsistency reflects the fundamental unreliability of asking an LLM to solve a constrained LP without access to a solver. In contrast, \modelname{}'s output is \emph{insensitive to input phrasing}: because the LLM's only role is to produce a structured session table before invoking the solver, re-ordering session descriptions or using informal time formats (``half past six'' vs.\ ``6:30\,pm'') produce the same schedule. Figure~\ref{fig:schedules} shows pixel-identical heatmaps and load profiles for 2019-06-15, confirming full optimizer parity from NL input.

\paragraph{Failure cases and limitations.}
On heavily congested days (e.g., 2018-11-05, 47 sessions), the 50~kW cap makes full service physically impossible. Both solver and \modelname{} leave 40--60~kWh unmet — an unavoidable consequence of the site constraint, not a system failure. The LLM baseline exhibits two distinct failure modes: catastrophic collapse (near-zero schedule, 0\% served) and energy over-spend (cost exceeds the uncontrolled charge-ASAP baseline). Our evaluation assumes clean, accurate session inputs and the agent has not been tested on adversarially corrupted or incomplete data.

\section{Conclusion}
% ============================================================

\modelname{} demonstrates that an LLM agent backed by a certified convex solver matches the numerical optimizer exactly on cost, peak load, unmet energy, and service rate — while accepting plain-language requests and returning verifiable explanations. The single-shot LLM baseline fails on both energy delivery and constraint satisfaction, confirming that tool access, not model capability, is the decisive factor. Limitations include single-site scope, fixed cap, and clean input assumptions. Future work includes persistent multi-turn session context, input validation for robustness, multi-site feeder constraints, carbon/peak objectives, and intra-day rolling replanning.
